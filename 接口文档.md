# 接口文档

## 1. 文档说明

- 项目：AI合同初审助手微信小程序 V1
- 用途：前后端真实联调契约
- 更新时间：2026-05-20

## 2. 当前后端状态总结

- 后端主链路已完成：上传/文本提交 -> 解析预览 -> 开始审核 -> 查询结果 -> 历史记录 -> 删除历史
- 当前技术栈：`Python 3.11 + FastAPI + SQLite`
- 文件解析稳定支持：文本型 `pdf`、常规 `docx`
- 审核能力当前仍为 Mock 审核器，但接口结构按正式版设计
- 当前联调默认地址：`http://127.0.0.1:8010`
- 健康检查路径：`GET /health`

## 3. 当前约束

- `doc` 文件只做降级提示，不承诺稳定解析
- OCR 不在 V1 范围内
- 暂未接入正式用户体系，联调阶段通过 `X-User-Id` 区分用户
- 审核结果仍为 Mock 逻辑，不代表真实模型能力

## 4. 最关键的联调支持点

- 后端统一返回 `success + data` 包裹结构
- 结果接口已同时返回：
  - `result`
  - `summary`
  - `risks`
  这样便于兼容前端原有 mock 页面结构
- 历史记录接口已补充更适合直接展示的字段：
  - `id`
  - `title`
  - `status_text`
  - `completed_at`
- 审核必须通过轮询 `/result` 获取最终状态

## 5. Base URL

本地联调默认使用：

```text
http://127.0.0.1:8010
```

如果后端实际启动端口变更，前端 `baseURL` 必须同步修改。

## 6. 请求头

联调阶段统一传：

```http
X-User-Id: demo-user
```

说明：

- 当前 V1 暂未接入登录态
- 后端通过 `X-User-Id` 绑定用户历史记录
- 前端联调阶段可先固定使用 `demo-user`

## 7. 统一返回结构

### 7.1 成功返回

```json
{
  "success": true,
  "data": {}
}
```

### 7.2 失败返回

```json
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "错误提示文案"
  }
}
```

前端建议：

- 先判断 `success`
- 失败时优先展示 `error.message`

## 8. POST /api/v1/contracts/upload

### 用途

上传合同文件并在后端完成解析。

### 请求方式

`multipart/form-data`

### 请求头

```http
X-User-Id: demo-user
```

### 请求参数

- `file`：合同文件

### 支持类型

- `pdf`
- `doc`
- `docx`

### 特别说明

- 当前 V1 稳定支持：文本型 `pdf`、常规 `docx`
- `doc` 会返回明确降级提示
- 扫描件 OCR 不在 V1 范围内

### 成功响应示例

```json
{
  "success": true,
  "data": {
    "task_id": "8c5f20d5f0d4448f8cb2a98ea362f6f4",
    "file_id": "8c5f20d5f0d4448f8cb2a98ea362f6f4",
    "file_name": "采购服务合同.docx",
    "upload_status": "uploaded",
    "parse_status": "parsed",
    "preview_text": "甲方：xxx\n乙方：xxx\n第一条 服务内容...",
    "char_count": 1268,
    "page_count": 3
  }
}
```

### 失败响应示例 1：类型不支持

```json
{
  "success": false,
  "error": {
    "code": "UPLOAD_UNSUPPORTED_TYPE",
    "message": "仅支持 PDF、DOC、DOCX 文件上传"
  }
}
```

### 失败响应示例 2：文件过大

```json
{
  "success": false,
  "error": {
    "code": "UPLOAD_FILE_TOO_LARGE",
    "message": "文件大小不能超过 10MB"
  }
}
```

### 失败响应示例 3：DOC 降级

```json
{
  "success": false,
  "error": {
    "code": "PARSE_UNSUPPORTED_DOC",
    "message": "当前 V1 暂不稳定支持 .doc 解析，请转换为 .docx 或直接粘贴文本"
  }
}
```

## 16. 联调最容易踩坑的点

### 16.1 服务地址必须确认

- 当前默认联调地址是 `http://127.0.0.1:8010`
- 如果前端访问 `/health` 返回 `404`，先确认当前 `8010` 端口是不是本项目后端
- 本项目正确探活路径就是：

```text
GET http://127.0.0.1:8010/health
```

### 16.2 文件上传不能走普通 JSON 请求

必须使用 `wx.uploadFile`。

### 16.3 结果接口已经做了兼容返回

为了兼容前端原有 mock 结构，后端现在同时返回：

- `result`
- `summary`
- `risks`

前端可以优先使用：

```text
data.summary
data.risks
```

### 16.4 历史记录接口已补展示字段

当前历史记录可直接使用：

- `id`
- `title`
- `status_text`
- `completed_at`

减少前端额外拼装。

### 16.5 字段风格仍以 snake_case 为主

后端主体字段仍为 `snake_case`，如果页面层想继续维持 camelCase，建议在请求层集中映射。

### 16.6 中文 `?` 需要区分“终端显示”与“接口真实返回”

- 使用不受 PowerShell 管道编码影响的 UTF-8 请求方式验证后，接口实际返回的中文是正常的
- 因此当前已知的 `?` 更像是终端脚本注入或控制台显示问题，不是后端 JSON 本身被破坏
- 仍建议在微信开发者工具界面中再做一次真实页面展示确认

## 17. 前端是否需要调整字段适配

需要，但成本已经下降很多。

建议：

- 预览接口继续在请求层做 `task_id -> taskId`、`parsed_text -> parsedText`
- 结果接口可直接消费 `summary` 和 `risks`，不用强依赖 `result`
- 历史接口可直接用 `id/title/status_text/completed_at`
