# 接口文档

## 1. 文档说明

- 项目：AI合同初审助手微信小程序 V1
- 用途：前后端真实接口联调契约
- 更新时间：2026-05-20

## 2. Base URL

本地开发默认使用：

```text
http://127.0.0.1:8010
```

如果启动时改了端口，前端 `baseURL` 也需要同步修改。

## 3. 请求头说明

联调阶段统一传：

```http
X-User-Id: demo-user
```

说明：

- 当前 V1 暂未接入正式登录态
- 后端通过 `X-User-Id` 绑定历史记录
- 联调时前端可先固定使用 `demo-user`

## 4. 统一返回结构

### 4.1 成功返回

```json
{
  "success": true,
  "data": {}
}
```

### 4.2 失败返回

```json
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "错误提示文案"
  }
}
```

前端联调建议：

- 先判断 `success`
- 若失败，优先展示 `error.message`

## 6. 上传文件接口

### 接口

```text
POST /api/v1/contracts/upload
```

### 请求方式

`multipart/form-data`

### 请求参数

- `file`：合同文件

### 支持类型

- `pdf`
- `doc`
- `docx`

### 说明

- 当前 V1 稳定支持文本型 `pdf` 与常规 `docx`
- `doc` 会返回降级提示，建议转成 `docx` 或改为粘贴文本
- 扫描件 OCR 不在 V1 范围

### 成功响应示例

```json
{
  "success": true,
  "data": {
    "task_id": "8c5f20d5f0d4448f8cb2a98ea362f6f4",
    "file_id": "8c5f20d5f0d4448f8cb2a98ea362f6f4",
    "file_name": "采购服务合同.docx",
    "upload_status": "uploaded",
    "parse_status": "parsed",
    "preview_text": "甲方：xxx\n乙方：xxx\n第一条 服务内容...",
    "char_count": 1268,
    "page_count": 3
  }
}
```

### 失败响应示例 1：格式不支持

```json
{
  "success": false,
  "error": {
    "code": "UPLOAD_UNSUPPORTED_TYPE",
    "message": "仅支持 PDF、DOC、DOCX 文件上传"
  }
}
```

### 失败响应示例 2：文件过大

```json
{
  "success": false,
  "error": {
    "code": "UPLOAD_FILE_TOO_LARGE",
    "message": "文件大小不能超过 10MB"
  }
}
```

### 失败响应示例 3：DOC 降级

```json
{
  "success": false,
  "error": {
    "code": "PARSE_UNSUPPORTED_DOC",
    "message": "当前 V1 暂不稳定支持 .doc 解析，请转换为 .docx 或直接粘贴文本"
  }
}
```

## 14. 联调最容易踩坑的点

### 14.1 统一包裹结构

真实后端数据在：

```text
response.data.data
```

不是直接平铺。

### 14.2 字段命名风格

后端当前使用 `snake_case`：

- `task_id`
- `source_type`
- `parsed_text`
- `file_name`
- `created_at`

前端现有 mock 使用 `camelCase`：

- `taskId`
- `sourceType`
- `parsedText`
- `fileName`
- `createdAt`

前端需要做字段映射。

### 14.3 结果结构差异

真实后端返回：

```text
data.result.total_risks
data.result.risks
```

前端 mock 当前读取：

```text
result.summary
result.risks
```

建议前端在请求层把真实接口转成页面现有结构。

### 14.4 审核必须轮询

- `startAudit` 只代表发起成功
- 前端需要轮询 `GET /result`
- `analyzing` 继续轮询
- `success` 进入结果页
- `failed` 展示 `error_message`

### 14.5 文件上传必须用 `wx.uploadFile`

不能把上传接口当普通 JSON 请求。

## 15. 前端适配建议

建议把适配逻辑收敛到 `miniprogram/services/api.js`，不要把真实接口原始结构散落到各页面。

推荐映射：

- 预览接口：`task_id -> taskId`，`parsed_text -> parsedText`
- 结果接口：把 `data.result` 转换成页面当前使用的 `summary + risks`
- 历史接口：把 `task_id` 转成 `id` 或 `taskId`

## 16. Mock 审核器切换真实模型的改造位置

主要改造文件：

- [audit_service.py](D:\HT_project\HT_project\backend\app\services\audit_service.py)
