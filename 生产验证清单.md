# 生产验证清单

更新时间：2026-05-20

## 1. 验证目标

- 验证真实大模型审核服务接入后，后端仍保持 V1 既有接口结构稳定
- 验证审核失败、超时、重试、降级流程可控
- 验证前端依赖的 `summary + risks` 兼容结构不受影响

## 2. 接口稳定性检查

- `GET /health` 返回 `{"success": true, "data": {"status": "ok"}}`
- `POST /api/v1/contracts/text` 仍返回 `task_id / status / preview_text / char_count`
- `POST /api/v1/contracts/{task_id}/audit` 仍返回 `status=analyzing`
- `GET /api/v1/contracts/{task_id}/result` 仍返回：
  - `status`
  - `error_code`
  - `error_message`
  - `result`
  - `summary`
  - `risks`
- `GET /api/v1/contracts/history` 仍返回：
  - `id`
  - `title`
  - `status_text`
  - `completed_at`

## 3. 真实模型调用检查

- 已配置 `OPENAI_API_KEY`
- `OPENAI_BASE_URL` 指向正确网关
- `OPENAI_MODEL` 配置明确
- 审核输入长度被限制，避免超长合同直接拖垮链路
- 模型输出能被解析为 JSON
- 模型输出缺字段时仍能被后端规范化

## 4. 稳定性检查

- 模型请求超时后能返回可理解错误
- 短暂失败时会自动重试
- 开启兜底时，真实模型失败后会降级到 mock 审核
- 关闭兜底时，真实模型失败后任务进入 `failed`
- 日志里能看到成功、失败、重试、降级事件

## 5. 结果质量检查

- 返回的 `total_risks / high_risks / medium_risks / low_risks` 计数正确
- 每个风险项都包含：
  - `title`
  - `level`
  - `reason`
  - `suggestion`
- `level` 只出现 `high / medium / low`
- `overall_message` 存在且可读
