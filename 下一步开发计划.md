# 下一步开发计划

## 1. 当前阶段判断

根据最新的 [开发进度.md](./开发进度.md)，项目当前已经完成：

- V1 前后端基线基本冻结
- 前端真实联调地址稳定为 `http://127.0.0.1:8010`
- 接口结构保持稳定
- DeepSeek provider 已接入并完成成功联网审核验证
- `result_formatter.py` 继续作为统一结构化输出层
- `/result` 与 `/history` 对前端兼容结构保持稳定
- 前端页面层已增加针对空风险、等级兼容、长耗时等场景的兜底
- 前端结果页、历史页、审核中页已整理为 UTF-8 版本
- 第二轮、第三轮中文合同样例验证已完成
- 中文结果展示链路整体判断正常
- DOCX 可读性主问题已明显收敛
- DeepSeek 调优已完成关键突破：
  - 使用 `deepseek-v4-flash`
  - 关闭 `thinking`
  - 调整超时、重试、backoff、token 上限
  - 真实模型返回耗时压缩到约 `4.1s`
  - 当前验证中未再出现 timeout
  - 当前验证中未再降级到 mock

项目当前仍未完成：

- 最终上线前生产部署形态收口
- 将当前临时启动方式改为长期运行方式
- 接入域名与 HTTPS
- 小程序切换到线上 API 域名
- 进行最后一轮线上生产验证

当前阶段结论：

项目已经从“DeepSeek 质量优化阶段”进入`部署与可上线收口阶段`。当前核心问题已不再是模型性能，而是如何把现有稳定版本正式部署为长期可访问的线上服务。

---

## 2. 下一步总目标

基于当前已稳定的 V1 版本，完成部署形态收口并准备正式上线验证。

建议顺序：

1. 固化后端长期运行方式
2. 接入域名与 HTTPS
3. 用 Nginx 反向代理 FastAPI
4. 小程序切到线上域名
5. 做最后一轮线上生产验证

---

## 3. 开发优先级

### P0：固化后端长期运行方式

目标：

- 让当前后端不再依赖手工 `uvicorn` 窗口运行

建议方案：

- 使用 `systemd` 管理 FastAPI 进程
- 保证开机自启
- 保证异常退出可自动拉起

验收标准：

- 重启服务器后服务可自动恢复
- 不再依赖人工 SSH 挂着窗口

---

### P1：接入域名与 HTTPS

目标：

- 为微信小程序正式调用准备合法线上地址

需要完成：

- 购买并解析域名
- 配置后端 API 二级域名，例如 `api.xxx.com`
- 配置 HTTPS 证书

验收标准：

- 可通过 `https://api.xxx.com/health` 访问健康检查
- 小程序后续可配置合法 request 域名

---

### P2：使用 Nginx 反向代理 FastAPI

目标：

- 不再直接暴露 `8010` 作为正式入口

建议结构：

```text
微信小程序
  ->
HTTPS 域名
  ->
Nginx
  ->
Uvicorn/FastAPI:8010
```

验收标准：

- 80/443 对外可用
- 8010 仅作为内部服务端口保留

---

### P3：小程序切换到线上 API 域名

目标：

- 从本地/裸 IP 联调切换到正式线上域名

需要完成：

- 前端 `baseURL` 切换为 HTTPS 域名
- 微信小程序后台配置合法 `request` 域名

验收标准：

- 小程序前端可通过 HTTPS 域名正常调用后端
- 不再依赖裸 IP 调试地址

---

### P4：执行最后一轮线上生产验证

目标：

- 确认当前版本具备试运行或小范围上线条件

建议覆盖：

1. 文本提交流程
2. PDF 上传流程
3. DOCX 上传流程
4. DeepSeek 成功审核
5. 超时/失败提示
6. 历史记录写入与读取
7. 删除历史记录
8. 中文结果展示

重点记录：

- 线上平均耗时
- 线上失败率
- 是否仍发生降级到 mock
- 中文结果可读性
- 前端展示稳定性

---

### P5：文档与编码整理

目标：

- 保持项目资料长期可维护

需要执行：

- 修复 [开发进度.md](./开发进度.md) 编码问题，统一为 UTF-8
- 确认 [接口文档.md](./接口文档.md)、[生产验证清单.md](./生产验证清单.md)、[部署方案.md](./部署方案.md) 编码正常
- 每轮开发完成后同步更新本文件与开发进度
