# AI合同初审助手 V1 部署方案

## 1. 文档目的

本文档用于说明 `AI合同初审助手` V1 的推荐部署路径、上线所需资源、部署步骤和后续扩展建议，帮助项目从本地开发环境进入可外部访问的试运行或正式运行阶段。

## 2. 当前项目形态

当前项目技术形态：

- 前端：微信小程序
- 后端：Python + FastAPI
- 模型服务：DeepSeek
- 数据存储：SQLite
- 当前联调方式：本地 `http://127.0.0.1:8010`

这意味着项目已经具备标准 Web 后端结构，因此最适合先采用`自有后端部署`方案，而不是为了上线重新改造成其他架构。

## 3. 推荐部署方案

### 3.1 推荐结论

V1 推荐优先采用：

`腾讯云轻量应用服务器 / CVM + 域名 + HTTPS + Nginx + Uvicorn`

这是当前最省事、最稳定、对现有代码改动最小的部署路线。

### 3.2 为什么推荐这条路线

- 现有 FastAPI 后端可以直接部署，不需要重构
- 小程序正式调用本来就需要 HTTPS 域名
- 运维逻辑清晰，便于排查
- 适合作为 V1 的试运行和早期正式版本
- 后续如果业务增长，再迁移到云托管或容器平台也比较顺滑

## 4. 不同部署路线对比

## 4.1 路线 A：轻量云服务器部署

特点：

- 成本较低
- 最容易上手
- 适合当前阶段
- 完全控制运行环境

适合当前项目的原因：

- 后端已成型
- Python 服务容易部署
- 便于配置域名、证书、环境变量和日志

## 4.2 路线 B：云托管 / 容器部署

特点：

- 更现代
- 更适合后续扩容
- 初次配置复杂度更高

适合未来阶段，不建议当前 V1 第一版先走这条路，除非你已经熟悉相关平台。

## 5. 推荐上线阶段策略

建议分两步：

### 第一步：先把 V1 跑起来

- 购买 1 台轻量云服务器
- 部署当前 FastAPI 后端
- 配置域名与 HTTPS
- 微信小程序切到线上域名联调

### 第二步：再评估是否升级架构

- 评估是否迁移 SQLite
- 评估是否迁移到容器平台或云托管
- 评估是否加用户鉴权、日志监控、任务队列等

## 6. 上线所需资源

V1 至少需要以下资源：

1. 一台公网服务器
2. 一个域名
3. HTTPS 证书
4. 微信小程序后台配置权限
5. DeepSeek 线上环境变量

## 7. 服务器建议

### 7.1 推荐类型

优先选择：

- 腾讯云轻量应用服务器
- 或腾讯云 CVM

### 7.2 初期配置建议

V1 小流量试运行建议：

- 2 核 CPU
- 2GB 或 4GB 内存
- 50GB 左右系统盘
- Ubuntu LTS

如果只是内部测试或小范围试运行，这个配置通常足够。

## 8. 域名与 HTTPS

### 8.1 域名

需要准备一个可备案、可解析到服务器公网 IP 的域名。

建议：

- 使用独立二级域名给后端 API
- 例如：`api.xxx.com`

### 8.2 HTTPS

微信小程序正式环境要求请求走合法 HTTPS 域名，因此你必须配置 SSL 证书。

推荐：

- 使用免费证书或云平台证书服务
- 由 `Nginx` 统一做 HTTPS 终止

## 9. 微信小程序侧配置

部署完成后，需要在微信小程序后台配置：

- 合法 `request` 域名

这里要注意：

- 正式环境不能长期直接请求本地地址
- 正式环境不能直接依赖裸 IP 方案
- 必须使用已配置好的 HTTPS 域名

## 10. 后端部署架构建议

推荐最小结构：

```text
微信小程序
    ->
HTTPS 域名
    ->
Nginx
    ->
Uvicorn / FastAPI
    ->
SQLite（当前阶段）
```

## 11. 后端部署步骤建议

## 11.1 服务器初始化

- 安装 Ubuntu LTS
- 更新系统包
- 安装 Python
- 安装 `pip`
- 安装 `venv`
- 安装 `nginx`

## 11.2 上传代码

可选方式：

- 通过 Git 拉代码
- 通过压缩包上传
- 通过 CI/CD 同步

推荐先用 Git 方式部署，最便于后续更新。

## 11.3 安装后端依赖

进入 `backend` 目录后：

- 创建虚拟环境
- 安装 `requirements.txt`

## 11.4 配置环境变量

至少需要配置：

- `AUDIT_PROVIDER=deepseek`
- `DEEPSEEK_API_KEY`
- `DEEPSEEK_BASE_URL`
- `DEEPSEEK_MODEL`
- 超时相关配置
- 重试相关配置
- 是否允许降级到 mock

## 11.5 启动 FastAPI

推荐使用：

- `uvicorn`
- 后续再配 `systemd` 或 `supervisor` 做进程托管

## 11.6 配置 Nginx

Nginx 负责：

- 监听 80 / 443
- 处理 HTTPS
- 转发到 FastAPI 服务端口

## 11.7 配置证书

- 绑定 HTTPS 证书
- 开启 TLS

## 11.8 小程序切换线上地址

将前端 `baseURL` 从本地：

`http://127.0.0.1:8010`

切换为线上：

`https://api.xxx.com`

## 12. 数据库建议

### 12.1 当前阶段

当前使用 SQLite：

- 小流量测试可接受
- 部署简单
- 成本低

### 12.2 后续建议

如果要正式运营或并发提升，建议迁移为：

- MySQL
- 或 PostgreSQL

原因：

- 并发能力更强
- 运维更可控
- 备份恢复更规范

## 13. 日志与监控建议

V1 至少建议保留：

- 请求日志
- 审核结果日志
- DeepSeek 调用失败日志
- 超时日志
- 降级到 mock 的日志

后续可增加：

- 错误告警
- 可用性监控
- 日志聚合

## 14. 安全建议

上线前至少注意：

- 不要把 DeepSeek key 写死在代码里
- 环境变量只放在服务器侧
- 限制服务器开放端口
- 保证 HTTPS 生效
- 不在日志里打印完整敏感合同正文

## 15. 当前阶段不建议做的事

- 不要为了部署先重构整套架构
- 不要一开始就上过重的微服务方案
- 不要为了 V1 先引入 OCR、企业协作、复杂权限体系
- 不要把 SQLite 长期当作正式生产数据库

## 16. 建议的落地顺序

推荐按这个顺序推进：

1. 购买轻量云服务器
2. 购买并解析域名
3. 部署 FastAPI 后端
4. 配置 DeepSeek 环境变量
5. 配置 HTTPS
6. 小程序配置合法域名
7. 小程序切换到线上 API
8. 做一轮线上联调
9. 做一轮线上生产验证

## 17. 最终建议

对当前项目，最合适的解决方案是：

先购买一台轻量云服务器，把当前 FastAPI 后端部署上线，再接域名和 HTTPS 给微信小程序调用。

这是 V1 成本最低、实现最快、与当前代码兼容性最高的路线。
